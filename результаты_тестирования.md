# Результаты тестирования

| Действия пользователя | Причина | Ожидаемое поведение |
|-----------------------|---------|----------------------|
| Хочу записаться на хилс к Настей Николаевой на 18:00 на Гоголя → бот: «На ближайшие дни занятий у Насти Николаевой не нашлось» | Расписание из CRM обрезалось по `CRM_BRANCH_ID` (только филиал Тест); занятия Насти на Гоголя не попадали в ответ. | |
| После доработок: «Здравствуйте! Хочу к Насте на хилз» → «Давайте на Гоголя» → бот: «На ближайшие дни занятий у Насти на Гоголя нет» | 1) В CRM имя преподавателя может быть «Анастасия Николаева»; проверка `"настя" in "анастасия николаева"` даёт False. 2) В ответе CRM у записей расписания может не быть `branch` или `branch.name`, из‑за чего фильтр по филиалу отбрасывает все записи. 3) Название филиала в CRM может отличаться от «Гоголя» в KB. | |
| Бот: «Ближайшее занятие у Кати на Гоголя» → пользователь: «Да, когда прийти?» → бот: «На ближайшие дни занятий у Кати на Гоголя нет» | **По логам уточнено.** В таблице `tool_calls`: каждый `get_schedule` возвращает `result: {count: 7}` — CRM всегда отдаёт 7 записей. В таблице `sessions` в слотах на момент второго ответа: `branch: "Гоголя"`, `teacher: "Катя"` (не «Кати»). Значит причина не в форме имени преподавателя. Причина: **фильтр по филиалу** в `generate_schedule_response`: после фильтрации по `branch_name` совпадению с «Гоголя» список `schedules` становится пустым (0 записей). Значит в ответе CRM у всех 7 записей либо нет `branch`/`branch.name`, либо название филиала не совпадает с «Гоголя». Первый ответ («занятие у Кати на Гоголя») — из контекста LLM по предыдущему результату инструмента (без фильтра по филиалу или с другими слотами). Второй ответ — вызов с branch=Гоголя, после фильтра по филиалу 0 записей → «нет занятий». | |
| «Запишите на гёрли» → «На Гоголя» → бот: «На ближайшие дни занятий по Girly Hip-Hop на Гоголя нет расписания» | Та же причина, что и выше: фильтр по филиалу в `generate_schedule_response`. При выборе «Гоголя» в слотах остаются только записи, у которых `schedule.branch_name` совпадает с «Гоголя». В ответе CRM у записей расписания либо нет `branch`/`branch.name`, либо название филиала другое — после фильтра 0 записей, бот отвечает «нет расписания» для любого направления (Girly Hip-Hop, High Heels и т.д.) на Гоголя. | |
| «Сколько стоит абонемент?» → бот дал список абонементов и цен. Дальше: «А пробное у вас есть?» → бот: «Пробное бесплатное! В каком филиале удобнее?» | Бот не уточнил, групповые или индивидуальные абонементы, и не показал полную сетку цен. Возможные причины: в промпте нет требования уточнять тип абонемента; в контекст из KB попадает ограниченный фрагмент; лимит длины ответа. **Дополнение:** на вопрос про пробное бот ответил «бесплатное», но сразу переключился на выбор филиала и выпал из контекста — не дал развёрнутый ответ о пробном (условия, как записаться, что входит), словно перевёл в сценарий записи вместо завершения темы про цены/пробное. | |
| «Хочу купить 2 абонемента по 8 занятий» → бот: «В каком филиале удобнее?» → «на Гоголя» → бот: «Отлично, на Гоголя! Какое направление вас интересует?» | По ощущению пользователя бот мог воспринять «на Гоголя» не как выбор филиала (например, как имя или неоднозначный ответ). В тексте ответа бот явно подставил филиал («Отлично, на Гоголя!»), но затем вёл в сценарий записи на занятие (выбор направления), а не в сценарий покупки абонементов (оформление, оплата, куда прийти). То есть контекст «хочу купить 2 абонемента» не распознан как намерение покупки — бот ведёт как при записи на занятие. | |
| «Давайте на филиале Тест» / «Так я же сказал, Тест» / «А есть филиал Тест?» → бот: «У нас нет филиала с названием Тест», предлагает Гоголя и др. | **Причина в коде.** В промпте блок «Наши филиалы» формируется в `prompt_builder._format_kb_context()` из `self._kb.branches` — это секция **`branches:`** в [knowledge/studio.yaml](knowledge/studio.yaml) (список BookingBranch: Семёновская, Гоголя, Чуркин, Алеутская). Филиал «Тест» описан только в **`studio.branches`** (studio: → branches:), но не входит в верхнеуровневую секцию `branches:`. В инструкции промпта сказано: «Называй ТОЛЬКО те филиалы, которые перечислены в "Наши филиалы" выше». LLM видит только 4 филиала и правильно отвечает, что «Теста» в списке нет. Чтобы бот предлагал «Тест», его нужно добавить в секцию `branches:` в studio.yaml либо формировать «Наши филиалы» из `studio.branches` (где Тест уже есть). | |
