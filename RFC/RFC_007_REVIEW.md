# Ревью RFC-007 v2: Post-Launch Quality Fixes

**Ревьюер:** Cursor (Auto)  
**Дата:** 5 марта 2026  
**Фокус:** решение багов, соответствие кодовой базе, архитектура, масштабируемость, мультитенантность.

---

## 1. Насколько RFC решает проблемы и баги?

**Плюсы:**
- Матрица сценарий → фиксы (раздел 7) связывает каждый из 12 провалов с конкретными фиксами — покрытие прозрачное.
- Приоритизация P0/P1/P2 и фазы A/B/C дают понятный порядок внедрения.
- Ограничение «token-neutral или token-negative» соблюдено: постоянные добавки ~+205 chars, условные по триггерам; worst case 4225 chars — в пределах разумного.

**Замечания:**
- **F1 (ask_service_exists):** В спецификации написано «Если триггер сработал — подставить список направлений», но в коде `_should_add_services_list` при `not slots.group` возвращает `False` и делегирует в `_add_phase_context`. При уже заполненном `slots.group` (сценарий 11 — «есть бачата») список не подставлялся именно из-за фазы. Триггер по леммам «есть», «направление» и т.д. это исправит, но условие «if not slots.group: return False» в RFC противоречит цели: при запросе «есть бачата» group уже может быть (High Heels), и мы как раз хотим подставить список. Нужно уточнить: подставлять список при срабатывании лемм **независимо** от `slots.group`, и не дублировать с _add_phase_context (в GREETING/DISCOVERY без group по-прежнему phase context).
- **F14 (G12):** В guardrails `check()` сейчас не принимает `session` или `recent_tools` из сессии. В RFC указано `session_context.get("recent_tools")` — в коде нужно либо передавать `slots.recent_tools` в `check()`, либо расширить сигнатуру. Поле `recent_tools` в SlotValues в RFC есть, но в спецификации G12 не сказано, откуда guardrail его получает (сейчас у check есть только `executed_tools` текущего хода). Нужно явно: «в engine при вызове _gr.check передавать merged set: executed_tools текущего хода ∪ slots.recent_tools».
- **F9 (gibberish):** Эвристика `len(set(t.replace(" ", ""))) <= 3` при `len(t) >= 4` может дать ложные срабатывания на короткие осмысленные реплики (например «ааа супер»). Имеет смысл порог поднять (например 5–6 символов) или требовать повторяющийся паттерн (одинаковые слоги), чтобы не резать эмоциональные сообщения.

**Итог:** Проблемы закрываются целенаправленно. Требуются уточнения по F1 (логика условия при `slots.group`), F14 (источник `recent_tools` в G12) и F9 (порог gibberish).

## 2. Соответствие архитектуре и кодовой базе

**Соответствует:**
- **F2:** `_parse_llm_response` в `engine.py` — место верное; fallback через regex и очистку JSON не трогает контракт CoreLLMResponse.
- **F3:** `cancel_flow.start()` и `transition_state(ConversationState.CANCEL_FLOW)` — в `cancel_flow.py` уже есть импорт `transition_state`; добавление одного вызова перед return согласовано с текущим FSM.
- **F5:** StyleResolver уже имеет `_lookup` (dict alias → list[ResolvedEntity]); добавление шага prefix match после exact/lemma — естественное расширение. Важно: в RFC в Step 3 написано `for alias, style in self._lookup.items()` — в реальности value это `list[ResolvedEntity]`, а не один style; нужно итерировать по ключам и при матче возвращать `self._lookup[alias]`.
- **F7:** `list_my_bookings` — логика через `_impulse.find_client` и `list_bookings` уже есть в `cancel_flow.py`; вынос в отдельный tool и вызов из engine согласован с текущим разделением (engine выполняет tool_calls, impulse — CRM).
- **F8:** Уведомление админа через `enqueue_outbound` — в кодовой базе уже есть канал отправки сообщений; нужен импорт/доступ к `enqueue_outbound` и к `settings.admin_telegram_chat_id` (config).
- **F13:** `_summarize_tool_result` в engine — сейчас модульная функция с (tool_name, result_text); RFC предлагает лимиты по tool_name — реализуемо без смены архитектуры.
- **F15:** Замена `final_message` при `missing` в блоке `parsed.intent == "booking"` — место в engine корректно; словарь `missing_names` можно вынести в константу.
- **SlotValues:** Добавление полей `recent_tools`, `gibberish_count` — модель с default и без strict; PostgreSQL JSONB при десериализации подхватит новые ключи (как уже сделано для escalation_pending_reason).

**Несоответствия / неточности:**
- **F3, direct return:** В RFC: «После execute_tool_calls проверить `if executed_tools & DIRECT_RETURN_TOOLS` и вернуть tool_results[tool_name]». Сейчас после execute идёт повторный вызов LLM (в цикле) и только потом `final_message`. Чтобы «вернуть результат инструмента напрямую», нужно не входить во второй LLM call при этих tool names — т.е. выйти из цикла по tool_calls и вернуть текст инструмента. При этом для `start_cancel_flow` нужно всё равно применить `slot_updates` из того же ответа LLM (если есть), иначе client_phone не попадёт в слоты. В RFC corner case упомянут, но в коде надо явно: apply slot_updates → then if start_cancel_flow in executed_tools → transition state уже в cancel_flow.start() → return tool result.
- **F14:** GuardrailRunner.check не принимает `recent_tools`. Нужно в engine перед вызовом check собирать `tool_names_for_g12 = executed_tools | set(slots.recent_tools or [])` и передавать в G12 именно этот set (или расширить сигнатуру check, чтобы G12 использовал «tools в этой сессии»). В текущем описании RFC ссылается на `session_context.get("recent_tools")` — в GuardrailRunner такого контекста нет, его нужно передать из engine.
- **F10 (2 направления):** В RFC для style_resolver предложен split по «и» и запятым. В resolve() возвращается `list[ResolvedEntity]`; при нескольких стилях engine должен получить несколько сущностей и «уточнить». Но сейчас слоты хранят один `group`. Для мульти-стиля либо нужен отдельный слот (например `groups: list[str]`) и явная логика в промпте «начни с первого», либо промпт без изменения слотов: «если видишь 2 направления в запросе — ответь уточнением». Второй вариант проще и не требует смены модели слотов — достаточно промпта (F10). Раздел F10 в RFC про split в resolve — опционален; минимально достаточно только промпта.

**Итог:** В целом RFC совпадает с архитектурой (engine ↔ guardrails, cancel_flow, retriever, resolver). Нужны уточнения: F3 (порядок apply slot_updates и direct return), F14 (откуда G12 берёт recent_tools), F5 (итерация по _lookup), F10 (достаточность только промпта без split в resolve).

## 3. Правильность архитектурного решения

**Сильные решения:**
- **Не подставлять список направлений всегда (F1), а по триггеру** — уменьшает размер контекста и приучает модель опираться на данные только когда вопрос про «что есть».
- **Sanitize JSON + извлечение message из partial (F2)** — правильный уровень: парсер остаётся в engine, контракт CoreLLMResponse не меняется, пользователь не видит сырой JSON.
- **Direct return для start_cancel_flow и escalate_to_admin (F3)** — ответ = результат инструмента, без второго LLM call. Это и дешевле, и предсказуемее.
- **Per-tool summary limits (F13)** — расписание получает больший лимит, остальные инструменты не раздуваются. Архитектурно ок.
- **G12 с учётом previous-turn tools (F14)** — проверка «упоминал время/день без вызова расписания» должна учитывать «расписание уже вызывали в этой сессии», иначе ложные срабатывания. Расширение контекста guardrail сессией — логично.

**Спорные:**
- **F9 (gibberish в engine до LLM):** Обработка до входа в LLM — плюс (экономия токенов). Минус: жёсткая эвристика в коде; при смене языка или другого типа клиента придётся править код. Для вертикального стартапа лучше вынести конфиг (порог длины, размер алфавита, исключения) в конфиг тенанта или feature flags.
- **F5 (prefix match в StyleResolver):** Для одной студии и 8–15 стилей — ок. При 50+ стилях prefix match может давать неоднозначности (несколько стилей с одинаковым префиксом). RFC это учитывает («if len(matches) > 1 return matches»). Для масштабирования позже можно заменить на scoring (prefix + длина совпадения) или конфигурируемые правила.

**Итог:** Архитектурно решения в основном верные. Эвристики (gibberish, отчасти prefix match) стоит заложить с возможностью конфигурации под тенанта/язык.

## 4. Есть ли решение проще, гибче и масштабируемее?

**Более простые альтернативы:**
- **F1:** Вместо отдельной категории `ask_service_exists` и нового метода можно в _add_phase_context всегда добавлять одну короткую строку «Направления: [список имён]. Если спрашивают о другом — скажи что нет.» не только при GREETING/DISCOVERY, но и при любом запросе, где в леммах есть «есть», «направление» и т.п. — т.е. один блок в retriever с условием по леммам, без дублирования с phase. Это проще, чем два места (phase + intent).
- **F16 (цены):** Поднять лимит до 400 для всех (как в RFC «или проще») — проще, чем вводить исключение по intent + «₽». Разница в 100 символов для типичного сообщения не критична.
- **F10 (2 направления):** Только промпт «если 2+ направления — уточни» без изменений в StyleResolver — проще и без риска поломать текущий resolve(); split по «и» можно добавить позже при реальных кейсах.

**Более гибкие:**
- **Эскалация (F8, F9):** Сейчас триггеры эскалации зашиты в промпт и частично в YAML. Для мультитенанта разумно: список триггеров эскалации + правило «при N gibberish подряд» хранить в tenant config (или в том же studio.yaml, но загружаемом по tenant_id). Тогда для новой студии не трогаем код.
- **F6 (policy триггеры):** Добавление «каблук», «обувь» — точечно. Более гибко: хранить policy_triggers в KB (studio.yaml) как список, а не хардкод в retriever; тогда каждая студия может добавить свои леммы.

**Более масштабируемые:**
- **Resolver (F5):** Prefix match — O(n) по числу алиасов. При росте числа стилей (50+) можно кэшировать нормализованные формы и использовать префиксный индекс (trie или sorted list + bisect). Пока n мало — текущее предложение ок.
- **Tool summary (F13):** Словарь _TOOL_SUMMARY_LIMITS по имени инструмента масштабируется: новые инструменты получают default лимит, при необходимости добавляется строка. Для мультитенанта лимиты можно вынести в конфиг (по tenant_id или глобально).

**Итог:** Часть фиксов можно упростить (F1 один блок по леммам, F16 глобальный лимит 400, F10 только промпт). Гибкость и масштаб лучше заложить через конфиг (эскалация, gibberish, policy triggers, tool limits), не хардкод в коде.

## 5. Не зашиваем ли мы логику одной студии?

**Риски зашивания:**
- **Русские леммы и триггеры:** F1 (`есть`, `бывать`, `проводить`, …), F6 (`каблук`, `обувь`), F9 (кириллица в regex), эскалация («позовите администратора», «жалоба») — всё ориентировано на русский и одну студию. При добавлении студии в Казахстане (русский + казахский) или только английский понадобятся отдельные наборы лемм/триггеров. **Рекомендация:** хранить триггеры (service_exists, policy, escalation) в KB или tenant config как списки строк; загрузка по языку/тенанту.
- **Жёсткие формулировки в промпте:** «Если спрашивают о другом — скажи что нет и предложи похожее», «Перенос записи = отмена + новая запись», «Заморозка абонемента: спроси причину и срок» — это текст одной студии. Для вертикального стартапа промпт лучше собирать из шаблонов или из tenant-specific блоков (как уже делается с KB), а не фиксировать русские фразы в коде.
- **F15 missing_names:** Словарь «branch → филиал», «group → направление» и т.д. — русский. При мультиязычности нужны подстановки по локали/тенанту.
- **Gibberish (F9):** Эвристика «да», «нет», «+», «-» — универсальна; порог по длине и разнообразию символов можно вынести в конфиг. Язык-специфичные правила (например, «что считается словом») — в конфиг по языку.

**Что уже не привязано к одной студии:**
- Структура слотов, инструментов, guardrails, CRM-адаптер — абстрактные. Конкретика (названия направлений, филиалов, тексты) в KB (studio.yaml). Добавление второй студии = новый YAML + tenant_id, без смены кода движка.
- F2, F3, F13, F14, F15 — по сути про поток данных и контракты; формулировки fallback (F2 «Уточните, пожалуйста, ваш вопрос») — единственное место, которое лучше брать из конфига/локали.

**Итог:** Часть RFC (триггеры, промпт-тексты, missing_labels) завязана на русский и одну студию. Для вертикального стартапа стоит: (1) вынести триггеры и подсказки в KB/tenant config, (2) собирать промпт из блоков по тенанту/языку, (3) оформить F9 и F15 с подстановкой строк из конфига. В рамках «починить одну студию сейчас» RFC допустим; перед масштабированием — рефакторинг конфигурируемых частей.

## 7. Рекомендации перед имплементацией

1. **F1:** Переписать условие `_should_add_services_list`: подставлять список при срабатывании лемм «есть»/«направление»/… даже если `slots.group` уже задан; в GREETING/DISCOVERY без group оставить текущий phase context.
2. **F14:** В engine при вызове `_gr.check` передавать в G12 объединённый set `executed_tools | set(slots.recent_tools or [])`; в GuardrailRunner расширить сигнатуру (например `recent_tools: set[str] | None = None`) и использовать в _g12.
3. **F3:** В коде после execute_tool_calls: применить slot_updates, затем если `start_cancel_flow` в executed_tools — выйти из цикла и вернуть результат инструмента (state уже переведён в cancel_flow внутри start()).
4. **F5:** В prefix match итерировать по `self._lookup` и при матче брать `self._lookup[alias]` (list), не «style».
5. **F9:** Ослабить условие gibberish (например порог 5–6 символов или проверка повторяющегося слога); заложить конфиг (порог, исключения) для последующего мультитенанта.
6. **Масштабирование:** В рамках этого RFC не менять контракт загрузки конфига; в описании или в отдельном «Follow-up» зафиксировать: триггеры (service_exists, policy, escalation) и тексты fallback вынести в KB/tenant config перед добавлением второй студии.

После этих правок RFC можно считать готовым к реализации по фазам A → B → C.
